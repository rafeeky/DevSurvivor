# Part 1: 게임 비전 & 개요

## 1.1 프로젝트 정보

| 항목 | 내용 |
|------|------|
| **제목** | Dev Survival |
| **장르** | 2D 캐주얼 서바이벌 (뱀서라이크) |
| **플랫폼** | 웹 브라우저 (HTML/CSS/JS) |
| **1판 플레이타임** | 3분 |
| **개발 형태** | AI 협업 프로토타입 실험 |

## 1.2 한 줄 정의

> AI가 점점 진화하는 환경 속에서, 플레이어는 개발자가 되어 3분 동안 살아남고 오늘의 작업을 끝내야 한다.

## 1.3 핵심 감정 리듬

| 구간 | 감정 |
|------|------|
| 초반 | "할 만한데?" |
| 중반 | "이제 좀 귀찮고 빡세네" |
| 후반 | "이건 진짜 나를 대체할 수도 있겠다" |
| 결말 | "오늘은 내가 이겼나, AI가 이겼나" |

## 1.4 개발 목적

- 사내 링크 공유용 HTML 프로토타입 제작
- 짧은 기간 안에 코어 루프 검증
- Claude Code + 서브 에이전트를 활용한 AI 협업 실험

## 1.5 성공 기준

### 게임 측면
- 3분 루프가 끊기지 않고 돌아간다
- 1판 종료 후 다시 플레이할 이유가 있다
- 처음 보는 사람도 컨셉을 30초 안에 이해한다
- 적의 진화가 설명 없이 직관적으로 읽힌다

### 실험 측면
- AI를 어디에 쓰면 효율적인지 체감한다
- 사람이 직접 붙들어야 하는 판단 지점이 보인다

# Part 2: 코어 루프 & 게임플레이

## 2.1 전체 게임 루프

```
로비
 └─ 시작하기
     └─ 플레이 (3분)
         ├─ 한 판 루프: 적 처치 → 경험치 → 레벨업 선택 → 스킬 강화 → 생존
         └─ 종료 (생존 or 사망)
             └─ 결과 정산
                 └─ 출시 포인트 획득
                     └─ 메타 업그레이드 (선택)
                         └─ 로비 복귀 → 재도전
```

## 2.2 조작 방식

| 입력 | 동작 |
|------|------|
| WASD / 방향키 | 이동 |
| 마우스 클릭 | 스킬 수동 발동 (액티브 스킬) |
| 스킬 단축키 **1/2/3/4** | 스킬 슬롯 1~4 발동 |
| (모바일) 가상 조이스틱 | 이동 |
| (모바일) 스킬 버튼 | 화면 우측 하단 배치 |

> **자동 공격 없음**: 모든 전투는 스킬 발동으로 처리한다.
> 기본 이동과 스킬 타이밍이 핵심 플레이.

## 2.3 3분 타임라인

| 구간 | 시간 | 등장 적 | 플레이어 목표 | 감정 |
|------|------|---------|-------------|------|
| 1 | 0:00 ~ 0:30 | 박스봇 | 조작 파악, 기본 생존 | "귀엽고 할 만하다" |
| 2 | 0:30 ~ 1:00 | 박스봇 + 카트봇 | 첫 압박, 첫 레벨업 | "좀 빨라졌네" |
| 3 | 1:00 ~ 1:40 | 카트봇 + PC봇 | 공간 관리 전환 | "이제 귀찮아지네" |
| 4 | 1:40 ~ 2:10 | 카트봇 + PC봇 혼합 | 빌드 체감, 실수 시 위기 | "버티기 쉽지 않다" |
| 5 | 2:10 ~ 2:35 | 미러봇 첫 등장 | 대체 감각 제시 | "저건 나 같은데?" |
| 6 | 2:35 ~ 2:50 | 미러봇 + 잔존 적 | 보스 전 최종 압박 | "마지막이 온다" |
| 7 | 2:50 ~ 3:00 | **AI봇** 등장 | 최종 러시 생존 | "결국 완성형이 나왔다" |

## 2.4 레벨업 타이밍 (권장)

| 레벨 | 권장 시점 |
|------|---------|
| 1 → 2 | 0:35 ~ 0:50 |
| 2 → 3 | 1:05 ~ 1:20 |
| 3 → 4 | 1:40 ~ 1:55 |
| 4 → 5 | 2:10 ~ 2:25 |
| 5 → 6 | 2:35 ~ 2:50 |

# Part 3: 플레이어 시스템

## 3.1 기본 스탯 (레벨 1 기준, 공통)

| 스탯 | 기본값 | 설명 |
|------|--------|------|
| **최대 HP** | 100 | 0이 되면 사망 |
| **이동 속도** | 220 px/s | 기본 이동 속도 |
| **피격 무적 시간** | 0.4초 | 피격 후 짧은 무적 |
| **스킬 슬롯** | 4개 | 1 / 2 / 3 / 4 |

> 세 캐릭터 모두 기본 스탯은 동일하다. 차이는 **고유 패시브 1개**로만 만든다.

## 3.2 체력 관리

- 기본 HP 회복 없음 (스킬로만 회복)
- HP 30% 이하 시 화면에 경고 연출 (붉은 테두리 점멸)
- 피격 시 넉백 없음 — 위치 유지, 대신 무적 프레임 제공

## 3.3 플레이어 비주얼

- 노트북을 든 캐릭터 실루엣 (측면 or 정면)
- 컬러: 파란 계열, 사람임이 직관적으로 보이는 형태
- 이동 시 간단한 스프라이트 애니메이션 (2~4프레임)

---

## 3.4 캐릭터 구성

Dev Survival은 **3종 캐릭터**를 제공하며, 각각 고유 패시브 1개로 플레이 방식을 차별화한다.
액티브 스킬은 공용 풀로 유지하고, 캐릭터별 패시브가 방향을 만든다.

### 캐릭터 목록

| 캐릭터 | 역할 타입 | 패시브명 | 한 줄 설명 |
|--------|---------|---------|-----------|
| 주니어 개발자 (남) | **속도형** | 빠른 적응 | 배우는 속도가 남다르다. 첫 성장 타이밍이 빠르다. |
| 시니어 개발자 (남) | **처리형** | 숙련된 디버깅 | 문제를 많이 보지 않는다. 핵심만 정확히 처리한다. |
| 주니어 개발자 (여) | **버티기형** | 끝까지 버티기 | 힘들수록 더 버틴다. 쉽게 무너지지 않는다. |

---

## 3.5 캐릭터별 고유 패시브

### 주니어 개발자 (남) — 속도형 / 빠른 적응

**컨셉**: 처음엔 약하지만 배우는 속도가 빠르다. 전장을 돌아다니며 경험치를 더 효율적으로 먹고, 첫 성장이 빨리 온다.

**효과**
- 경험치 획득량 증가
- 첫 레벨업 필요 경험치 감소
- 이동속도 소폭 증가

**플레이 차이**: 초반 빌드 형성이 가장 빠름. 생존보다는 "빨리 성장해서 위험 구간을 넘기는" 타입. 박스봇/카트봇 구간에서 강함.

**밸런스 의도**: 초반이 편한 대신, 직접적인 화력/방어 보너스는 적다.

---

### 시니어 개발자 (남) — 처리형 / 숙련된 디버깅

**컨셉**: 한 번에 문제의 핵심을 잡는다. 강한 적, 귀찮은 적, 후반 고위협 적을 더 안정적으로 처리한다.

**효과**
- 단일 대상 피해 증가
- 엘리트/보스 대상 추가 피해
- `디버그` 계열 스킬 효율 증가

**플레이 차이**: PC봇, 미러봇, AI봇 대응력이 높음. 화면을 싹 정리하는 타입보다 "핵심 위협 제거"에 특화. 후반 안정성이 좋음.

**밸런스 의도**: 초반 다수 처리 능력은 평범하지만, 위험한 적을 빨리 잘라내는 능력으로 난이도를 낮춘다.

---

### 주니어 개발자 (여) — 버티기형 / 끝까지 버티기

**컨셉**: 압박이 와도 무너지지 않고 버틴다. 회복, 보호막, 피해 완화 같은 안정성에 강점이 있다.

**효과**
- 회복량 증가
- 보호막/자동 저장 효율 증가
- 체력 낮을 때 피해 감소 또는 회복 보정

**플레이 차이**: 실수가 나도 복구가 쉬움. 초보자 친화적. 긴장 구간, 미러봇 진입 구간에서 특히 강함.

**밸런스 의도**: 화력은 낮을 수 있지만, 위험 구간을 넘기는 능력이 가장 좋다.

---

## 3.6 캐릭터 선택 화면 표시 항목

플레이어가 "누가 더 센가"보다 **어떤 방식으로 살아남을 것인가**를 선택하게 만든다.

- 캐릭터 픽셀 아트
- 이름
- 역할 타입 라벨 (버티기형 / 처리형 / 속도형)
- 고유 패시브명
- 짧은 설명 1줄

# Part 4: 적 시스템

## 4.1 적 진화 계보

```
박스봇 → 카트봇 → PC봇 → 미러봇 → AI봇
(허술)   (반응)   (방해)  (대체)    (완성)
```

## 4.2 적 스탯 표

| 이름 | HP | 이동 속도 | 충돌 데미지 | 처치 경험치 |
|------|-----|---------|------------|-----------|
| 박스봇 | 30 | 80 px/s | 10 | 5 |
| 카트봇 | 55 | 130 px/s | 15 | 10 |
| PC봇 | 80 | 70 px/s | 12 | 15 |
| 미러봇 | 150 | 170 px/s | 25 | 30 |
| AI봇 (보스) | 600 | 200 px/s | 35 | 200 |

## 4.3 적 행동 패턴 상세

### 박스봇
- 플레이어 방향으로 직진
- 특수 패턴 없음
- 다수 동시 스폰 (군집형)

### 카트봇
- 플레이어 추적
- 3초마다 짧은 돌진 (속도 2배, 0.6초간)
- 방향 전환 빠름

### PC봇
- 플레이어와 거리 유지 (150~250px)
- 2.5초마다 **오류탄** 발사 (느린 투사체, 명중 시 2초간 이동속도 -30%)
- 6초마다 **방해 구역** 생성 (반경 80px, 4초간 지속, 진입 시 데미지)

### 미러봇
- 빠른 근접 추적
- 4초마다 짧은 연속 돌진 (3회 연속)
- 플레이어 방향으로 예측 이동

### AI봇 (최종 보스)
- 고속 추적 (기본)
- 5초마다 강한 돌진 (광역 1회)
- 8초마다 **충격파** (플레이어 중심 200px 원형, 데미지 + 넉백)
- 체력 50% 이하 시 **하위 적 소환** (카트봇 2마리, 20초마다)

## 4.4 스폰 규칙

| 구간 | 스폰 간격 | 동시 최대 적 수 |
|------|---------|--------------|
| 0:00 ~ 0:30 | 2.5초 | 5 |
| 0:30 ~ 1:00 | 2.0초 | 8 |
| 1:00 ~ 1:40 | 1.8초 | 10 |
| 1:40 ~ 2:10 | 1.5초 | 12 |
| 2:10 ~ 2:35 | 2.0초 (미러봇 포함) | 8 |
| 2:35 ~ 2:50 | 2.0초 | 10 |
| 2:50 ~ 3:00 | AI봇 단독 등장 (나머지 스폰 중지) | — |

- 스폰 위치: 화면 바깥 랜덤 방향에서 등장
- 동시 최대 적 수 초과 시 스폰 대기

# Part 5: 스킬 & 판 내 성장

## 5.1 경험치 & 레벨업

| 레벨 | 필요 경험치 (누적) |
|------|----------------|
| 1 → 2 | 30 |
| 2 → 3 | 80 |
| 3 → 4 | 160 |
| 4 → 5 | 280 |
| 5 → 6 | 450 |

- 레벨업 시 게임 일시 정지 후 **3지선다** 제시
- 선택지: 신규 스킬 획득 / 기존 스킬 강화 / 기본 능력 강화
- 레벨 6 이후에는 스킬 강화만 제공

## 5.2 스킬 목록

### 전투형 스킬

| 스킬 | 역할 | 기본 수치 | 쿨다운 |
|------|------|---------|--------|
| **긴급 수정** | 근거리 범위 공격 | 반경 120px, 데미지 40 | 5초 |
| **디버그** | 단일 고화력 | 가장 가까운 적 1명, 데미지 70 | 2.5초 |
| **우선순위 정리** | 전방 부채꼴 + 넉백 | 부채꼴 120°/160px, 데미지 30, 넉백 100px | 7초 |

### 생존/회복형 스킬

| 스킬 | 역할 | 기본 수치 | 쿨다운 |
|------|------|---------|--------|
| **자동 저장** | 보호막 (피격 1회 무효) | 12초마다 자동 갱신 | 12초 |
| **커피 한 잔** | 단기 버프 | 이동속도 +40%, 쿨다운 -20%, 5초 지속 | 20초 |
| **산책하기** | 이동속도 강화 | 이동속도 +30% (지속) | 패시브 (or 액티브 8초) |
| **피규어 청소하기** | 피해 감소 + 넉백 | 받는 피해 -40%, 주변 150px 넉백, 4초 지속 | 12초 |
| **강아지 쓰다듬기** | 회복 | 최대 HP의 25% 회복 | 20초 |
| **낮잠자기** *(확장)* | 고위험 고회복 | 1.5초 정지 후 HP 50% 회복, 모든 쿨다운 -3초 *(미구현 — 추후 levelup.js에서 처리)* | 30초 |

> `낮잠자기`는 MVP에서 제외 가능한 확장 스킬.

## 5.3 스킬 강화 (레벨업 선택지)

각 스킬은 최대 3단계 강화 가능.

| 스킬 | 강화 1 | 강화 2 | 강화 3 |
|------|--------|--------|--------|
| 긴급 수정 | 데미지 +20 | 반경 +40px | 쿨다운 -1.5초 |
| 디버그 | 데미지 +30 | 관통 (2명) | 쿨다운 -0.8초 |
| 자동 저장 | 갱신 주기 -3초 | 보호막 2회 | 피격 시 주변 소폭 데미지 |
| 커피 한 잔 | 지속 +3초 | 버프 효과 +10% | 쿨다운 -3초 |
| 강아지 쓰다듬기 | 회복량 +10% | 회복 중 피해 -20% | 쿨다운 -5초 |

# Part 6: 경제 & 메타 성장

## 6.1 스코어 공식

```
최종 점수 = 생존 점수 + 처치 점수 + 진행률 점수 + 보너스
```

| 항목 | 계산식 |
|------|--------|
| 생존 점수 | 생존 초 × 10 (최대 1800점) |
| 기본 적 처치 | 처치 수 × 5 |
| PC봇 처치 | 처치 수 × 10 |
| 미러봇 처치 | 처치 수 × 30 |

# Part 8: 아트 & 연출

## 8.1 전체 비주얼 방향

- **스타일**: 미니멀 픽셀 or 플랫 벡터 (구현 난이도 우선)
- **배경**: 현대 오피스 내부 — 책상, 모니터, 파티션, 유리창, 형광등
- **색상 키**: 플레이어(파랑), 박스봇(연갈색), 카트봇(복고 주황), PC봇(회색), 미러봇(은색), AI봇(검정+발광)
- **맵 세계관**: "야근 중인 회사" — 사무실 단면도, 창문 너머 도시 스카이라인

---

## 8.1b 오피스 맵 방향

### 메인 플레이 맵 컨셉

**오피스 내부 / 야근 중인 회사**

| 구역 | 설명 |
|------|------|
| 상단 | 큰 유리창 + 도시 스카이라인 배경 |
| 중단 | 책상/모니터가 놓인 실무 구역 |
| 하단 | 이동 공간이 더 넓은 복도/업무 동선 |

### 타일셋 방향

- 오피스 바닥 타일
- 유리창 벽, 책상/회의실/복도
- 서류 더미 / 모니터 / 서버랙
- 간단한 휴게 공간 요소

**시각 우선순위**
1. 플레이 공간이 명확해야 한다
2. 오브젝트가 적/플레이어 식별을 방해하면 안 된다
3. 회사 생활 압박을 읽히게 하는 오브젝트가 중요하다

### 오브젝트 카테고리

| 카테고리 | 오브젝트 |
|---------|---------|
| 배경형 | 큰 유리창, 도시 스카이라인, 천장 조명, 벽면 프레임 |
| 필드형 | 책상, 의자, 파티션, 서류 더미, 모니터, 휴지통, 커피컵, 화분 |
| 상황 연출형 | 빨간 느낌표, 불꽃 효과, 오류창 패널, 긴급 알림 패널 |

### 오브젝트 배치 원칙

- 플레이 가능한 동선을 먼저 확보
- 시각적 정보는 상단/배경 쪽에 더 많이 배치
- 충돌 오브젝트는 최소화
- 장식형 오브젝트는 맵 외곽에 우선 배치
- "업무 압박" 오브젝트는 이벤트/적 등장 구간에서 강조

### 시각 연출 요소

| 상태 | 연출 |
|------|------|
| 평상시 | 모니터 빛, 서류 더미, 형광등, 커피컵 |
| 압박 구간 | 느낌표, 불꽃, 오류창 패널, 경고 배너 |
| AI 진화 구간 | 차가운 색감, 화면 깜빡임, 기계적 UI 패널, 푸른 발광 |

## 8.2 적 비주얼 진화 방향

| 단계 | 외형 느낌 | 색상 | 크기 |
|------|---------|------|------|
| 박스봇 | 허술한 박스 로봇 | 연갈색 | 작음 |
| 카트봇 | 복고풍 게임기 | 주황/노랑 | 중간 |

> 핵심: 같은 "로봇 계열"이 업그레이드되는 흐름으로 보여야 함.
> 서로 다른 종족이 아니라 **하나의 AI가 발전하는 과정**.

# Part 10: Agent Collaboration Architecture

> 이 파트는 **Dev Survival의 AI 협업 실험** 구조를 정의한다.
> 결과물(게임) 제작과 실험(AI 역할 분담 검증)이 동시에 진행되며,
> 이 섹션은 그 두 목적을 모두 지원하는 운영 설계서다.

## 10.1 Subagent Usage Purpose

### 왜 서브 에이전트를 사용하는가

이 프로젝트는 단일 AI에게 전체 작업을 위임하는 방식이 아니라,
**역할이 명확히 분리된 서브 에이전트들이 각자의 도메인을 담당하고
메인 에이전트가 통합·조정하는 구조**를 실험한다.

### 기대하는 효과

| 목적 | 설명 |
|------|------|
| **병렬 처리** | 독립적인 모듈(적 시스템, 스킬 시스템 등)을 동시에 개발 |
| **컨텍스트 집중** | 각 에이전트가 자신의 도메인 GDD만 인식해 출력 품질 향상 |
| **책임 추적** | 어떤 에이전트가 무엇을 만들었는지 명확히 기록 가능 |
| **실험 데이터** | 역할 분리가 실제로 효율적인지 검증 가능 |

### 이 구조로 검증하고 싶은 핵심 질문

- 메인 AI 1개 vs 역할 분리된 서브 에이전트, 어느 쪽이 더 빠르고 정확한가?
- 어떤 작업이 AI에 잘 맞고, 어디서 사람이 개입해야 하는가?
- 핸드오프 과정에서 정보 손실이 얼마나 발생하는가?

## 10.2 Main Agent Integration Role

### 역할명: Integration Lead

메인 에이전트는 특정 모듈을 직접 구현하지 않는다.
**서브 에이전트의 산출물을 수신하고, 통합하고, 최종 판단을 내리는 책임**을 가진다.

### 핵심 책임

| 책임 | 설명 |
|------|------|
| **태스크 분배** | GDD 파트를 기반으로 각 서브 에이전트에 작업 지시 |
| **산출물 수신 & 검토** | 각 에이전트 output이 검수 기준을 충족하는지 확인 |
| **충돌 해결** | 에이전트 간 수치·구조 충돌 시 최종 조율 |
| **통합 빌드 조율** | 각 모듈이 하나의 게임으로 합쳐질 수 있도록 인터페이스 정의 |
| **인간 판단 지점 관리** | 감정 리듬, 밸런스 피드백, 공유 판단 등 AI가 결정하지 않는 항목 명시 |
| **실험 로그 위임** | 모든 협업 과정을 Fleeky에게 기록 위임 |

### 메인 에이전트가 결정하지 않는 것

- 게임의 감정 리듬이 실제로 잘 작동하는지 (플레이 피드백 필요)
- 아트 스타일 최종 선택
- 공유 대상 및 배포 시점

사람 (사용자)
  │
  ▼
[메인 에이전트: Integration Lead]
  │
  ├─── GDD Part 2, 3, 9 ──────► Raf (Core System Designer)
  │                                │ 완료 시 player.js, game.js 전달
  │                                ▼
  ├─── GDD Part 4 + player.js ─► Marv (Enemy & Combat Designer)
  │                                │ 완료 시 enemies.js, spawner.js 전달

  ├─── GDD Part 5 + game.js ───► Haon (Skill & Progression Designer)
  │                                │ 완료 시 skills.js, levelup.js 전달
  │                                ▼
  ├─── GDD Part 6, 7 ──────────► Mason (UI/UX Flow Designer)
  │                                │ 완료 시 hud.js, lobby.js, result.js 전달

  │    [메인 에이전트: 통합 빌드]
  │      모든 모듈 연결 검토
  │      인터페이스 충돌 조율
  │      최종 index.html 통합
  │                                │
  ├─── 변경 사항 발생 시 ────────► Milli (GDD Editor)
  │                                │ GDD.md 업데이트
  │                                ▼
  └─── 전 과정 관찰 ─────────────► Fleeky (Experiment Logger)
                                   │ experiment_log.md 기록
                                   ▼
                            [실험 종료: 인사이트 문서 발행]

### 핸드오프 규칙

| 규칙 | 설명 |
|------|------|
| **의존성 순서 준수** | Raf → Marv / Haon 순서로 진행 (player.js 선행 필요) |
| **Mason은 병렬 가능** | UI는 게임 로직과 독립적으로 진행 가능 |
| **Milli는 상시 대기** | 설계 변경이 발생하는 즉시 호출 |
| **Fleeky는 silent observer** | 작업을 막지 않고 관찰만 수행 |
| **메인 에이전트 승인 없이 통합 금지** | 각 모듈은 메인 에이전트가 검토 후 통합 |
| **QA 전담 에이전트 없음 (프로토타입 한정)** | 각 에이전트가 자신의 검수 기준을 직접 수행, 메인 에이전트가 통합 최종 검토 |

### 사람이 반드시 판단해야 하는 지점

| 판단 지점 | 이유 |
|----------|------|
| 감정 리듬이 실제로 느껴지는가 | 수치로 검증 불가, 플레이 체감 필요 |
| 적 진화가 직관적으로 읽히는가 | 아트 + 움직임의 복합 판단 |
| 첫 공유 시점 결정 | 사내 맥락 고려 |
| 승패 연출 최종 선택 | 블랙코미디 톤 유지 여부 |
| 밸런스 최종 조정 | 플레이 피드백 기반 |

## 10.5 QA 구조 결정 사항

> **결정**: 현재 프로토타입 단계에서는 별도의 QA 전담 에이전트를 두지 않는다.

### 프로토타입 단계 QA 구조

| 검증 주체 | 범위 |
|---------|------|
| **각 서브 에이전트** | 자신의 산출물에 대한 검수 기준을 context 파일에 보유하고 직접 수행 |
| **메인 에이전트 (Integration Lead)** | 통합 단계에서 모듈 간 연결 및 전체 루프 최종 검토 |

### QA 전담 에이전트 확장 조건

아래 중 하나에 해당할 때 QA 에이전트(역할명 미정) 추가를 검토한다:

- 플레이테스트 범위가 확대되어 체계적인 회귀 검증이 필요해질 때
- 에이전트 수 또는 모듈 수가 늘어나 메인 에이전트 단독 통합 검토가 어려워질 때
- v1.0 이후 반복 배포 구조가 생길 때

## 10.6 AI Collaboration Hypotheses to Validate

이 섹션은 실험의 목적을 가설 형태로 명시한다.
각 가설은 프로토타입 완료 후 Fleeky의 로그를 통해 검증된다.

> **참고**: QA 전담 에이전트 없음 (10.5 참조). 검증 데이터는 Fleeky가 수집한다.

| # | 가설 | 검증 방법 | 예상 결과 |
|---|------|---------|---------|
| H1 | 역할이 명확한 작업(스킬 수치 구현 등)은 에이전트가 단독으로 완성할 수 있다 | Haon 산출물 재작업 여부 측정 | 재작업 1회 이하 |
| H2 | 핸드오프 과정에서 수치 충돌이 발생한다 | Milli의 충돌 기록 카운트 | 2~3회 발생 예상 |
