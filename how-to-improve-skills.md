# How to Improve Claude Skills for **Dev Survival**
부제: **GDD 중심 서브 에이전트 협업 품질을 자동으로 끌어올리는 운영 가이드**

---

## 문서 목적

이 문서는 **Dev Survival** 프로젝트에서 Claude Code 스킬과 서브 에이전트를 더 안정적으로 운용하기 위한 실전 가이드다.  
일반적인 “프롬프트를 잘 쓰는 법”이 아니라, 현재 작성된 GDD와 Agent Collaboration Architecture를 기준으로 아래를 개선하는 데 목적이 있다.

- GDD 기반 산출물의 **일관성 향상**
- 서브 에이전트별 역할 수행 품질 개선
- 핸드오프 시 정보 손실 감소
- 반복 실행 시 결과 흔들림 감소
- AI가 잘하는 일 / 사람이 직접 판단해야 하는 일의 경계 명확화

이 문서는 특히 **Part 10: Agent Collaboration Architecture**를 실제로 운영 가능한 개선 루프로 연결하기 위해 작성한다.

---

## 1. Why This Matters

지금 Dev Survival은 단순히 게임을 만드는 프로젝트가 아니다.  
이 프로젝트는 동시에 다음을 실험한다.

1. **Claude Code + Subagent 분업 구조가 실제 제작에 유효한가**
2. **역할 분리된 에이전트들이 GDD를 얼마나 정확히 실행할 수 있는가**
3. **반복 테스트를 통해 각 에이전트용 스킬을 자동 개선할 수 있는가**

즉, 스킬은 한 번 잘 나오면 끝나는 도구가 아니라,  
**반복 실행하면서 점점 안정화되는 작업 프로토콜**로 다뤄야 한다.

---

## 2. What “Improving Skills” Means in This Project

이 프로젝트에서 “스킬 개선”은 문장을 예쁘게 다듬는다는 뜻이 아니다.  
아래처럼 **실제 제작 성공률을 올리는 것**을 의미한다.

- 같은 입력을 줬을 때 GDD 해석이 흔들리지 않는가
- 필요한 항목이 누락되지 않는가
- 산출물이 담당 역할 범위를 벗어나지 않는가
- 수치, 구조, 파일 책임이 명확한가
- 다른 에이전트에게 넘겨도 충돌이 적은가
- 메인 에이전트가 통합할 때 수정 비용이 줄어드는가

즉, 중요한 기준은 **“실제 제작에 바로 쓰일 수 있는가”**다.

## 3. Core Principle: Small Change, Re-Test, Keep or Revert

기본 원리는 단순하다.

1. 현재 스킬로 결과물을 만든다
2. 체크리스트로 점검한다
3. 낮은 점수의 원인을 찾는다
4. 스킬 프롬프트를 **한 군데만 작게 수정**한다
5. 다시 실행한다
6. 좋아졌으면 유지하고, 나빠졌으면 되돌린다
7. 반복한다

핵심은 **한 번에 크게 바꾸지 않는 것**이다.  
작게 바꿔야 무엇이 실제로 좋아졌는지 추적할 수 있다.

## 4. Where to Apply This in Dev Survival

Dev Survival에서 자동 개선 루프를 적용하기 좋은 대상은 아래와 같다.

### A. GDD 해석 스킬
예:
- 특정 Part만 읽고 구현 단위로 분해하기
- GDD의 수치를 코드 구조로 변환하기
- Part 간 의존성 찾기

### B. Subagent Role Execution Skill
예:
- Raf가 Part 2, 3, 9를 기반으로 어떤 파일까지 만들어야 하는지 해석
- Marv가 Part 4만 보고 enemies.js / spawner.js 책임을 명확히 지키는지 점검
- Mason이 UI 구현 시 Part 6과 7의 경계를 정확히 구분하는지 확인

### C. Handoff Summary Skill
예:
- “내가 만든 산출물 / 다음 에이전트가 알아야 할 인터페이스 / 아직 미정인 것”을 짧게 넘겨주기

### D. GDD Update Skill
예:
- 실제 구현 변경사항을 Milli가 GDD에 누락 없이 반영하는지 검증

### E. Experiment Logging Skill
예:
- Fleeky가 작업 로그를 사람이 사내 공유할 수 있는 수준으로 요약하는지 확인

## 5. What to Optimize First

모든 스킬을 한 번에 개선하려고 하지 말고, 아래 순서로 최적화하는 것이 좋다.

### 1순위
**Subagent Role Execution Skill**

이유:
- 지금 프로젝트의 핵심 목적이 “서브 에이전트 실험”이기 때문
- 역할이 흔들리면 이후 산출물 전체가 흔들림
- 핸드오프 품질과 통합 비용에 직접 영향

### 2순위
**Handoff Summary Skill**

이유:
- Part 10 구조상 에이전트 간 의존성이 있음
- 정보 손실이 가장 자주 일어나는 지점이 핸드오프임

### 3순위
**GDD Update Skill**

이유:
- 실제 구현과 문서가 어긋나면 이후 모든 테스트 기준이 무너짐

### 4순위
**Experiment Logging Skill**

이유:
- 프로젝트 목적상 인사이트 문서가 중요하지만, 게임 구현보다 우선순위는 한 단계 낮음

## 6. The Checklist Method

자동 개선 루프에서 가장 중요한 건 **“좋은 결과가 무엇인지”를 yes/no 체크리스트로 정의하는 것**이다.

체크리스트는 3~6개가 적당하다.  
너무 많으면 에이전트가 형식만 맞추는 쪽으로 과적응할 수 있다.

### 좋은 체크리스트의 조건
- 명확하다
- 예/아니오로 답할 수 있다
- 역할 범위를 직접 검증한다
- 실제 핸드오프/통합 문제를 잡아낸다

---

## 7. Example Checklists for This Project

### 7.1 Raf (Core System Designer) 개선용 체크리스트
- 결과물이 **Part 2, 3, 9만** 근거로 작성되었는가?
- `index.html`, `game.js`, `player.js` 책임이 명확히 구분되었는가?
- 게임 루프, 타이머, 플레이어 이동이 모두 언급되었는가?
- 적/스킬 구현 책임을 Raf가 침범하지 않았는가?
- 다음 핸드오프 대상(Marv, Haon)이 필요한 인터페이스가 명시되었는가?

### 7.2 Marv (Enemy & Combat Designer) 개선용 체크리스트
- 적 5종이 GDD Part 4 스탯 표와 일치하는가?
- 패턴 구현 설명이 각 적별로 분리되어 있는가?
- `enemies.js`와 `spawner.js` 책임이 나뉘어 있는가?
- 2:50 이전에는 AI봇이 등장하지 않는다고 명시했는가?
- 플레이어 로직이나 UI 구현 책임을 침범하지 않았는가?

### 7.3 Haon (Skill & Progression Designer) 개선용 체크리스트
- 경험치 커브와 레벨업 흐름이 Part 5와 일치하는가?
- 스킬 목록이 현재 MVP 범위와 일치하는가?
- 스킬별 수치와 쿨다운이 빠지지 않았는가?
- `skills.js`와 `levelup.js` 책임이 구분되어 있는가?
- 메타 성장 영역(Part 6)을 과도하게 침범하지 않았는가?

### 7.4 Mason (UI/UX Flow Designer) 개선용 체크리스트
- 로비 / HUD / 레벨업 / 결과 / 업그레이드 화면이 모두 반영되었는가?
- 화면 흐름이 로비 → 인게임 → 결과 → 업그레이드/재시작으로 자연스러운가?
- UI 설명이 GDD Part 7 와이어프레임과 일치하는가?
- localStorage 관련 책임이 명시되어 있는가?
- 게임 로직을 직접 구현하려 들지 않았는가?

### 7.5 Milli (GDD Editor) 개선용 체크리스트
- 변경사항이 어느 Part에 반영되어야 하는지 명시했는가?
- 구현 변경과 GDD 수치 차이를 추적 가능한 형식으로 정리했는가?
- 버전, 날짜, 변경 내용이 기록되었는가?
- 선택지 형태로 조율안을 제시했는가?
- 구현되지 않은 상상 요소를 문서에 임의 추가하지 않았는가?

### 7.6 Fleeky (Experiment Logger) 개선용 체크리스트
- 각 에이전트별 입력/출력/시간/재작업 여부가 기록되었는가?
- 사람 개입 지점과 이유가 기록되었는가?
- “AI에 적합한 작업 / 사람 판단이 필요한 작업” 구분이 있는가?
- 로그가 사내 공유 가능한 문체로 정리되었는가?
- 단순 서술이 아니라 실험 인사이트까지 도출했는가?

## 9. What Good Looks Like for This Project

이 프로젝트에서 좋은 산출물은 아래 특징을 가진다.

- **근거가 되는 GDD Part가 명확하다**
- 역할 범위를 넘지 않는다
- 출력 파일/산출물 단위가 분명하다
- 다음 에이전트가 바로 이어받을 수 있다
- 메인 에이전트가 통합하기 쉽다
- 사람이 나중에 다시 봐도 의도가 추적 가능하다

즉, “좋은 말”보다 **좋은 인터페이스**가 중요하다.

## 10. How to Run the Improvement Loop

### Step 1. 고칠 스킬을 하나 고른다
예:
- Raf role execution skill
- Marv enemy implementation skill
- Handoff summary skill
- GDD update skill

### Step 2. 테스트 입력 세트를 고른다
가능하면 3개 정도를 묶는다.  
한 입력에만 최적화되면 실제로는 약한 스킬이 될 수 있다.

### Step 3. 체크리스트를 3~6개 만든다
각 역할별로 yes/no 평가가 가능해야 한다.

### Step 4. 현재 스킬로 baseline을 측정한다
각 테스트 입력에 대해 결과물을 만들고 체크리스트 점수를 합산한다.

### Step 5. 프롬프트를 한 군데만 수정한다
예:
- “근거 GDD Part를 반드시 먼저 명시하라” 추가
- “담당하지 않는 파일은 절대 제안하지 마라” 추가
- “출력 형식을 입력 / 출력 / 검수 기준 표로 고정하라” 추가

### Step 6. 다시 실행하고 비교한다
점수가 좋아졌으면 유지, 나빠졌으면 되돌린다.

### Step 7. 반복한다
목표는 완벽이 아니라 **재작업률 감소**다.

## 11. What Changes Usually Help

이 프로젝트 기준으로 잘 먹힐 가능성이 높은 수정은 보통 아래 유형이다.

### A. 근거 문서 범위 고정
예:
- “반드시 Part 4만 사용해서 답하라”
- “Part 6과 7 외 내용은 가정하지 마라”

### B. 역할 침범 금지 문장 추가
예:
- “UI Designer는 게임 로직을 설계하지 않는다”
- “Enemy Designer는 메타 성장 수치를 바꾸지 않는다”

### C. 출력 형식 고정
예:
- 입력 / 출력 / 검수 기준 / 미정 사항 / 다음 핸드오프

### D. 금지 규칙 추가
예:
- GDD에 없는 시스템을 임의 제안하지 말 것
- 구현 범위를 늘리지 말 것
- MVP 외 확장안을 기본안처럼 섞지 말 것

### E. worked example 추가
짧은 좋은 예시를 붙이면 형식 안정성이 높아진다.

## 12. What Usually Fails

### 1) 범위를 넓혀버림
역할이 작은데도 “더 좋은 게임”을 만들겠다며 범위를 확장하는 경우

### 2) 근거 없는 추정 추가
GDD에 없는 적 패턴, 스킬, 파일 구조를 임의 추가하는 경우

### 3) 문장만 그럴듯하고 구현 단위가 없음
책임 파일, 데이터 구조, 상태 전환이 빠지는 경우

### 4) 핸드오프 정보 부족
다음 에이전트가 알아야 할 인터페이스가 없는 경우

### 5) 체크리스트에 과적응
형식만 맞추고 실제로는 내용이 빈약해지는 경우

## 13. Recommended Prompt Constraints

Dev Survival용 스킬에는 아래 제약을 넣는 것이 유효하다.

- **근거 GDD Part를 먼저 명시할 것**
- **담당하지 않는 영역은 다루지 말 것**
- **출력은 구현 단위로 나눌 것**
- **다음 핸드오프 대상이 있으면 입력 인터페이스를 적을 것**
- **GDD에 없는 기능은 “제안”과 “확정”을 구분할 것**
- **MVP 범위를 넘는 확장은 별도 섹션으로 분리할 것**

---

## 14. Suggested Score Formula

간단한 점수 체계를 하나 두면 자동 비교가 쉬워진다.

예시:

- 체크리스트 5개 중 통과 수 × 20점
- 역할 침범 발생 시 -20점
- GDD 외 임의 추가 발생 시 -20점
- 핸드오프 정보 명시 시 +10점

### 예시
- 5개 중 4개 통과 = 80점
- 역할 침범 1회 = -20점
- 최종 점수 = 60점

이런 식으로 baseline과 개선 결과를 비교할 수 있다.

---

## 15. Suggested Stop Conditions

자동 개선 루프를 언제 멈출지 기준도 필요하다.

### 추천 종료 조건
- 3회 연속 90점 이상
- 같은 수정 유형이 2회 연속 효과 없음
- 체크리스트는 통과하지만 실제 구현 품질이 좋아지지 않음
- 메인 에이전트가 더 이상 구조적 병목이 아니라고 판단

## 16. How This Connects to Part 10 of the GDD

이 문서는 GDD Part 10을 실제 운영 가능한 형태로 보조한다.

### Part 10에서 정의한 것
- 누가 무엇을 맡는가
- 어떤 산출물을 내야 하는가
- 어떻게 핸드오프하는가
- 어떤 가설을 검증하는가

### 이 문서가 추가하는 것
- 각 역할의 스킬을 **어떻게 반복 개선할 것인가**
- 무엇을 기준으로 품질을 측정할 것인가
- 작은 수정을 어떤 방식으로 유지/폐기할 것인가

즉, Part 10이 **협업 구조 정의서**라면,  
이 문서는 **협업 품질 개선 운영 가이드**다.

가장 먼저 돌려볼 실험은 아래를 추천한다.

### 대상 스킬
**Marv — Enemy & Combat Designer skill**

- 출력 품질을 yes/no로 평가하기 쉽다

- enemies.js / spawner.js 책임이 분리되었는가
- AI봇 등장 시점이 2:50 이후로 고정되었는가
- UI / 메타 성장 역할을 침범하지 않았는가

## 19. Human Review Points

아무리 자동화해도 아래는 사람이 끝까지 봐야 한다.

- 감정 리듬이 실제로 느껴지는가
- 적의 진화가 직관적으로 읽히는가
- 승패 연출 톤이 블랙코미디로 작동하는가
- 첫 플레이에서 30초 안에 컨셉이 이해되는가
- 사내 공유용으로 적절한가

즉, 자동 개선 루프는 **구조와 품질의 안정화**에는 강하지만,  
최종 체감 판단까지 대신하지는 않는다.

## 20. Final Takeaway

이 프로젝트에서 스킬 개선의 핵심은 멋진 프롬프트를 한 번에 만드는 것이 아니다.  
진짜 중요한 건 다음이다.

- 역할을 작게 나눈다
- 좋은 결과의 기준을 yes/no 체크리스트로 만든다
- 한 번에 한 군데만 수정한다
- 테스트하고 유지/폐기한다
- 기록을 남긴다

Dev Survival에서 Claude Code를 잘 쓴다는 건  
단순히 코드를 빨리 뽑는다는 뜻이 아니다.  
**GDD를 기준으로 역할을 분리하고, 각 역할의 품질을 반복적으로 안정화하는 운영 능력**까지 포함한다.
